Parcours d’intégration Web Audio (composants réutilisables)

Ce document présente une démonstration guidée : enchaînement, dans une page hôte, des mêmes briques que le lecteur complet (<my-audio-player>), avec branchement manuel sur un graphe Web Audio dont l’état se construit à chaque étape. L’AudioContext n’est pas créé à l’étape 1 : il est obtenu via integration-demo-host.js au premier branchement Web Audio (étape 3), pour laisser la lecture HTML5 native fonctionner sans contrainte CORS inutile. Pas de new AudioContext dans le HTML.

Éléments attendus pour l’évaluation
  • Un seul AudioContext pour toute la chaîne ; il est passé aux composants via setAudioContext(ctx) lorsque la spécification du composant l’exige (EQ, WAM).
  • La playlist et le composant <my-player-controls> ne créent pas de nœuds Web Audio : ils émettent des événements ou reçoivent des branchements que le script hôte relie au média et aux GainNode / StereoPannerNode.
  • L’ordre des étapes ci-dessous est la séquence pédagogique attendue : contrôle (contrat controls-*), entrée Web Audio, traitement (EQ, WAM optionnel), mesure (analyseur + visualiseur). Si l’étape 2 n’est pas cliquée, le montage de <my-player-controls> se fait au premier clic sur l’étape 3. Le VU du bloc contrôles ne reflète le signal qu’après l’étape analyseur.
  • Pour réinitialiser la démonstration, un rechargement de la page est requis : le navigateur n’autorise qu’une MediaElementAudioSourceNode par élément <audio>.

Ordre à suivre (numérotation affichée sur chaque carte)

  1. Playlist + <audio> — Source de vérité des pistes ; lecture HTML5 possible avant toute connexion Web Audio.
  2. <my-player-controls> — Montage du composant et abonnement aux événements controls-* ; volume sur HTMLMediaElement.volume tant que le graphe n’est pas branché ; balance sans effet jusqu’à l’existence d’un StereoPannerNode. Le même montage peut être déclenché automatiquement au clic sur l’étape 3 si l’étape 2 n’a pas encore été validée.
  3. MediaElementSource → gain maître → panoramique → destination — À partir d’ici, le volume et la balance du bloc contrôles pilotent les nœuds Web Audio ; audio.volume est fixé à 1 pour éviter une double atténuation.
  4. <my-equalizer> — Insertion avec le pont GainNode recommandé dans la SPECIFICATION (éviter de passer le MediaElementSource directement à setInput si d’autres branches existent).
  5. <wam-plugin> (facultatif) — Avant l’analyseur dans ce scénario ; après analyseur, ce parcours linéaire refuse l’insertion WAM.
  6. Analyseur + <my-visualizer> — L’analyseur est placé sur la queue du signal avant le gain maître dans cette démo ; le VU du bloc contrôles lit ensuite ce nœud.

Dès qu’au moins une piste est présente, l’indicateur d’étapes avance sur la carte « Étape 2 » (avant le montage de <my-player-controls>). L’étape 1 (playlist + média) est prête dès l’ouverture avec les pistes par défaut. Utilisez Recharger la page seulement pour réinitialiser tout le graphe Web Audio après createMediaElementSource (une seule source par balise audio). Documentation : SPECIFICATION.md et README.md.

Étape 1 — Playlist et élément média

<my-playlist> tient la liste et notifie les changements. L’élément <audio> est le média réel : tant qu’aucun MediaElementAudioSourceNode n’a été créé, le son sort par la sortie « native » du navigateur. Les pistes par défaut viennent de l’attribut data-tracks sur la playlist et du script hôte ; ouvrez la page via un serveur HTTP local (npx http-server) pour que les fichiers sous assets/ se chargent. Pas d’attribut crossorigin sur audio à l’étape 1 (évite un blocage si le serveur n’envoie pas d’en-têtes CORS).

Étape 2 — <my-player-controls>

Le composant est défini dans playercontrols.js et importe localement webaudiocontrols.js pour le knob (consigne cours : pas de script webaudio-controls dans le HTML). Il s’agit de monter le tag, puis d’écouter les événements composés (controls-play, controls-volume, etc.) pour les relier à audio et à la playlist. Tant que l’étape 3 n’est pas effectuée, le volume agit sur audio.volume ; la balance n’a pas encore de cible dans le graphe. Le bouton de l’étape 3 s’active dès qu’il y a au moins une piste et qu’aucun MediaElementSource n’a été créé : si vous n’avez pas cliqué ici, le montage de <my-player-controls> et le branchement des événements se font au premier clic sur « Brancher le média au graphe ».

Étape 3 — Entrée dans Web Audio

createMediaElementSource attache définitivement ce graphe à cette balise audio. On crée un GainNode (volume maître) et un StereoPannerNode (balance), puis on relie au destination du contexte. Les valeurs affichées sur le bloc contrôles sont recopiées sur ces nœuds ; audio.volume est ramené à 1 pour que seul le gain Web Audio contrôle le niveau.

Schéma après clic : MediaElementSource → GainNode → StereoPanner → destination.

Étape 4 — <my-equalizer>

L’égaliseur reçoit le même AudioContext. Le média est déconnecté du gain maître, passé dans un GainNode pont, puis setInput / getOutput selon l’API du composant. La sortie de l’EQ revient vers le gain maître déjà relié au panoramique et à la sortie.

Étape 5 — <wam-plugin> (optionnel)

Un effet WAM peut être inséré entre la sortie de l’EQ et le gain maître. Si l’étape 6 est enclenchée sans étape 5, cette branche est considérée comme sautée une fois le visualiseur actif. Après insertion de l’analyseur, ce scénario linéaire n’autorise plus l’ajout du WAM sans rechargement de la page.

Étape 6 — Analyseur et <my-visualizer>

Un AnalyserNode est inséré sur la queue courante avant le GainNode maître dans ce montage pédagogique. setAnalyser + start() sur le visualiseur affichent le spectre ou la forme d’onde. Le VU du bloc contrôles (étape 2) peut alors être alimenté par des échantillons temporels issus de ce même analyseur lorsque la lecture est active.
