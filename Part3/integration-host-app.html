Du zéro à la chaîne — composants utilisables dans n’importe quel contexte

Les composants ne créent pas votre monde audio à votre place : vous fournissez le AudioContext (le vôtre, celui d’un framework, d’un onglet partagé, d’un module chargé à part…), puis setAudioContext / setInput / setAnalyser. Ils restent de simples HTMLElement branchés sur votre graphe.

Un flux = un même AudioContext. Ce n’est pas une limite des Web Components : c’est la façon dont la Web Audio API relie les nœuds. L’« indépendance », ici, veut dire : aucune dépendance à notre lecteur — import du seul fichier du composant, puis API explicite — et libre choix de l’origine du contexte (voir étape 0).

Recharger la page pour tout remettre à zéro (un seul MediaElementSource par <audio>).

Étape 0 — D’où vient le AudioContext ? (ce n’est pas la même chose en code)

Les deux boutons aboutissent au même type d’objet (AudioContext), mais pas au même rôle dans votre architecture :

Option A — Votre page est le propriétaire du contexte Le script de cette page appelle explicitement new AudioContext(). Typique d’une appli qui contrôle tout le son.
Option B — Le contexte est créé ailleurs (module « hôte ») Ce fichier ne contient aucun new AudioContext. Il importe integration-demo-host.js, qui simule une autre couche (bundle tiers, worker loader, autre équipe…). La page ne fait qu’obtenir une référence : même objet Web Audio, mais contrat d’intégration différent — comme dans une vraie appli modulaire.
